# Migration Rules

## Core policy

- Use raw SQL migrations with Alembic wrapper files.
- Do not commit ORM-autogenerated migration files.
- Each migration must be a small, coherent change.
- Prefer additive, forward-safe schema changes.

## Required shape

For migration `0001`:

- `db/migrations/versions/0001.py`
- `db/migrations/sql/0001.sql`

**Filenames are bare numeric** — `0001.sql`, `0002.sql`, `0003.sql`, etc. No descriptive suffix (not `0001_init.sql`, not `0002_documents_and_chunks.sql`). The Alembic wrapper derives its `revision` id from the filename, so the SQL file and the Python wrapper must share the same numeric stem. Describe what the migration does in the SQL comment header and the batch doc, not in the filename.

The Python wrapper should only locate and execute the SQL file.

When a migration also adds SQL objects, keep them in dedicated files:

- `db/migrations/sql/views/<name>_view.sql`
- `db/migrations/sql/functions/fn_<name>.sql`

## SQL rules

- Use lowercase snake_case names.
- Use explicit constraints.
- Add indexes intentionally, not later by accident.
- Use `timestamp with time zone default now()` for system timestamps.
- Use `text primary key` when the app generates string IDs.
- Keep DDL readable and reviewable.

## Dual-maintenance rule

The repo keeps two synchronized representations of the schema:

1. `db/migrations/versions/NNNN.py` — Alembic scripts, used for incremental upgrades.
2. `db/migrations/sql/NNNN.sql` — raw SQL mirror, used by Docker Compose to initialize a fresh database in one shot.

Both files share the same bare numeric stem (e.g. `0002.py` ↔ `0002.sql`). No descriptive suffix on either side.

When you add a migration:

1. Write the Alembic script first.
2. Run `bash db/migrations/migrate.sh` to verify it applies cleanly.
3. Hand-copy the equivalent SQL into `db/migrations/sql/NNNN.sql`.
4. Tear down the Docker volume and rebuild to verify the init path also works.

Skipping step 3 silently breaks fresh installs. CI does not catch it. Reviewers must.

## Views

- Keep each view in its own file under `db/migrations/sql/views/`.
- Start with `drop view if exists <name>;` or `drop view if exists <name> cascade;` when dependencies require it.
- Then define the view with `create or replace view <name> as`.
- Prefer one clear select statement per view file.
- Use view files for read models and JSON aggregation patterns, not for hiding business rules that belong in actions.

## Functions

- Keep each function in its own file under `db/migrations/sql/functions/`.
- Name functions with an `fn_` prefix.
- Use `create or replace function`.
- Use explicit argument types and explicit return types.
- Add `comment on function ...` for non-trivial functions so the contract of the function stays readable.
- Prefer SQL functions where possible; use `plpgsql` when control flow is actually needed.

## Wrapper rules for views and functions

- The Alembic version file may execute multiple SQL files after the main migration SQL.
- Keep the version file as a loader, not as the place where SQL objects are authored.
- If the main SQL file contains `\ir` include lines for readability, strip those lines in the wrapper before executing it through SQLAlchemy, then execute the referenced files explicitly.
- Downgrades must explicitly drop views and functions by name and full function signature where required.

## Object layout pattern

Use this sequence when a migration adds a table plus a view or function:

1. create or alter base tables in `db/migrations/sql/<version>.sql`
2. execute that file from `db/migrations/versions/<version>.py`
3. execute any related view files
4. execute any related function files
5. keep downgrade logic in the wrapper limited to explicit drop statements

## Vector and full-text index rules

When a table will be queried by retrieval:

- Add an HNSW index on the embedding column with `vector_cosine_ops` (or `vector_ip_ops` if the embedding model is normalized).
- Add a GIN index on a `tsvector` column generated from the searchable text columns.
- Both indexes belong in the initial migration that creates the table, not a follow-up migration. Adding them later forces an index build on a populated table, which blocks writes.

## Safety rules

- Avoid destructive operations unless explicitly approved.
- Avoid silent data rewrites inside schema migrations.
- For non-trivial backfills, prefer a separate, explicit migration step.
- If a migration depends on existing data shape, document that assumption in the SQL comments.

## Review checklist

- Does the migration match the current batch?
- Is the SQL idempotent where practical?
- Are indexes and uniqueness constraints included?
- Are timestamps and nullability explicit?
- Is the wrapper file only loading SQL, not hiding logic?
- Are views and functions in dedicated files instead of embedded into the wrapper?
- Do downgrade statements cover every created view and function?
- Has the SQL mirror in `db/migrations/sql/` been updated?
