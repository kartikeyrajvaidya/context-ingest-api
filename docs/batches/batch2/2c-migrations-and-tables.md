# Batch 2c — Database schema and migrations ⏳

**Goal:** Turn the contracts from 2b (as re-frozen 2026-04-14 for conversation support) into concrete tables, indexes, and SQLAlchemy models. Produce the real migrations and model bodies. After this batch, `psql` shows the full v0 schema and models can be imported and used, but no action yet reads or writes them (that's 3a for documents/chunks and 3b/3c for query_requests / query_responses / feedback).

**Scope:** this batch turns the frozen 2b contracts into concrete tables and ORM models. All design decisions are called out explicitly below.

**Baseline:** 2a originally shipped `0001_init.sql` as an empty placeholder. When the filename was renamed to `0001.sql` (bare numeric), the "init" suffix was the only reason the placeholder existed — so 2c now **overwrites `0001.sql` with real content**. No deployed instances need to worry about the change: the project is pre-release, no one has run the old empty migration in production.

**Migration file naming.** Bare numeric: `0001.sql`, `0002.sql`, `0003.sql`, etc. No descriptive suffix in the filename — content is described in the SQL comment header and the batch doc. The Alembic wrapper derives its `revision` id from the filename (`0002.py` → `revision = "0002"`), so renaming files is the only thing that changes version ids.

**Files added — migrations:**
- `db/migrations/sql/0001.sql` — **overwritten**. Enables `vector` extension, creates `documents` and `chunks` tables with all indexes. Split from 0002 so a future batch (or fork) can run just the ingestion half without the query/response/feedback tables.
- `db/migrations/versions/0001.py` — Alembic wrapper. `down_revision = None` (first revision). No more empty-file guard — 0001 now has real content.
- `db/migrations/sql/0002.sql` — **new**. Creates `query_requests`, `query_responses`, and `feedback` tables with FKs + indexes. Owns the whole "queries + feedback" half of the schema.
- `db/migrations/versions/0002.py` — Alembic wrapper. `down_revision = "0001"`.

**Files filled in (stubs from 2a):** each currently contains only `__tablename__`, an `id` column, and `get_id_prefix()`. 2c adds the real column definitions and helpers.

- `db/models/base.py` — touch only if we decide to share a timestamp mixin.
- `db/models/documents.py` — flesh out `Document` with all columns from the table spec below, plus helpers: `get_by_source_url(session, url)`, `find_active(session)`.
- `db/models/chunks.py` — flesh out `Chunk` with all columns, using `tsv = Column(TSVECTOR, Computed("to_tsvector('english', chunk_text)", persisted=True))` — generated column declared on the ORM side so the DDL lives with the model. Helpers: `delete_by_document_id`, `count_by_document_id`, `hybrid_search` (RRF via raw SQL).
- `db/models/query_requests.py` — flesh out `QueryRequest` with all columns and a `conversation_belongs_to_session(session_id, conversation_id)` classmethod used by 3b to decide whether to echo or re-mint.
- `db/models/query_responses.py` — flesh out `QueryResponse` with all columns plus a `fetch_recent_completed_turns(session_id, conversation_id, limit)` classmethod that joins `query_requests` and orders by `created_at DESC`, limited to `CONVERSATION_HISTORY_TURN_LIMIT`, reversed to chronological. Used by 3b to build the prompt transcript block.
- `db/models/feedback.py` — flesh out `Feedback` with all columns, plus an `upsert_by_query_id(session, query_id, rating, reason)` classmethod wrapping the Postgres `ON CONFLICT (query_id) DO UPDATE` statement. FK target is `query_requests.id`. The action in 3c calls the classmethod; no route-level SQL.

**Frozen table shapes:**

All `id` columns are `text` PKs of the shape `<prefix><random10>` — e.g. `doc_a3f8k2m9p1`. Generated by `BaseModel.create` in `db/models/base.py` using the existing `_generate_random_string(10)` helper (lowercase alphanumerics, 36^10 ≈ 3.6 × 10¹⁵ combinations per prefix). Readable in logs, short enough to paste into a terminal, and collision-safe at v0 scale. Any model without a registered `get_id_prefix()` is a bug.


`documents` — one row per logical source.
| column | type | notes |
|---|---|---|
| id | `text` PK | `doc_<random10>` (e.g. `doc_a3f8k2m9p1`) |
| source_url | `text` **not null, unique** | For raw-text ingestion the CLI provides a caller-supplied canonical identifier like `internal://faq-setup`. Never nullable — every document has a stable identifier. |
| source_type | `text` not null | `url`, `text`, future: `pdf`, `notion`. |
| title | `text` nullable | Falls back to HTML `<title>` or a derived slug. |
| content_hash | `text` | SHA-256 of cleaned content. Used by rerun-safety check. |
| chunk_count | `integer` not null default 0 | Convenience mirror; updated inside the same txn as chunk insert. |
| is_active | `boolean` not null default true | Soft-disable without deleting embeddings. |
| last_ingested_at | `timestamptz` default `now()` | |
| created_at | `timestamptz` default `now()` | |

Indexes: `source_type`, `is_active`, `created_at`, unique on `source_url`. *(No `authority` column — domain-specific authority tagging is out of scope for a generic RAG base.)*

`chunks` — one row per embedded chunk.
| column | type | notes |
|---|---|---|
| id | `text` PK | `chk_<random10>` |
| document_id | `text` FK `documents.id` **on delete cascade** | Cascade keeps "reingest" simple: delete the doc row and children fall with it. |
| chunk_order | `integer` not null | |
| chunk_text | `text` not null | |
| embedding | `vector(1536)` not null | |
| tsv | `tsvector` generated always as `to_tsvector('english', chunk_text)` stored | |
| created_at | `timestamptz` default `now()` | |

Indexes: `document_id`, HNSW on `embedding` with `vector_cosine_ops`, GIN on `tsv`, `created_at`. Unique on `(document_id, chunk_order)` — guards against double-insert during retries.

`query_requests` — one row per inbound `/v1/query` call, written **before** the retrieval + LLM pipeline runs. The PK is the public `query_id` returned on the response and referenced by `feedback.query_id`. *(The request/response split keeps request data in tidy columnar form separate from the richer response, so operators who fork the project can add their own response shape without touching the request table.)*

| column | type | notes |
|---|---|---|
| id | `text` PK | `qry_<random10>`. Exposed to callers as `query_id`. |
| question | `text` not null | |
| session_id | `text` not null | Caller-supplied. Required on every request. |
| conversation_id | `text` not null | Server-minted (`cnv_<random10>`) if the caller omitted or supplied a value that doesn't belong to `session_id`. Never null. |
| created_at | `timestamptz` default `now()` | |

Indexes: `created_at`, `session_id`, `conversation_id`, composite `(session_id, conversation_id, created_at DESC)` for the conversation-history fetch.

`query_responses` — one row per **completed** `/v1/query` call, written after the pipeline finishes (success or failure). Joined 1:1 to `query_requests` via `request_id`. A request with no matching response row is a dangling/crashed request and is excluded from history fetches.

| column | type | notes |
|---|---|---|
| id | `text` PK | `qrs_<random10>`. Internal — never exposed to callers. |
| request_id | `text` **not null, UNIQUE** FK → `query_requests.id` **on delete cascade** | 1:1 relation with requests. UNIQUE enforces the 1:1 and makes it the natural lookup key from the response side. |
| session_id | `text` not null | **Denormalized** from the request so response-only analytics queries don't need a join. |
| conversation_id | `text` not null | Denormalized for the same reason. |
| status | `text` not null | `ok` or `failed`. No `pending` — rows are written only after the pipeline returns. |
| answer | `text` nullable | Null on no-answer or `status='failed'`. |
| citations | `jsonb` not null default `'[]'` | **Full snapshot** of the citations returned to the caller: `document_id, chunk_id, chunk_order, source_url, title, text, ingested_at`. Stored as JSONB so (a) historical answers survive re-ingestion of the source document, and (b) operators forking the project can add citation fields (`authority`, `source_type`, etc.) without a schema migration. |
| confidence | `text` nullable | `high \| medium \| low`. Matches the 2b contract. |
| retrieved_chunk_ids | `text[]` not null default `'{}'` | Eval replay. May be a superset of chunk_ids referenced in `citations` if some retrieved chunks were pulled but not cited. |
| next_actions | `jsonb` not null default `'[]'` | Suggested follow-up prompts returned to the caller (see 2b). Stored as JSONB so historical replays can see what was offered without re-running the LLM. Empty array on no-answer or malformed LLM output. No count cap at the DB level — the upper bound is a prompt-layer rule, not a storage rule. |
| created_at | `timestamptz` default `now()` | |

Indexes: `created_at`, `status`, `session_id`, `conversation_id`.

**Two-write flow (Batch 3b, documented here for cross-reference):**

1. Validate request. Mint `conversation_id` if caller omitted it or `QueryRequest.conversation_belongs_to_session(session_id, conversation_id)` returns `false`.
2. `INSERT INTO query_requests ...` and commit. The row is now durable and its `id` is the `query_id` eventually returned to the caller.
3. Fetch prior turns via `QueryResponse.fetch_recent_completed_turns(session_id, conversation_id, CONVERSATION_HISTORY_TURN_LIMIT)`. Skipped if `conversation_id` was just minted.
4. Run retrieval + LLM generation.
5. `INSERT INTO query_responses ...` with `status='ok'` (or `'failed'` on exception) and commit.
6. Return the full response envelope.

A crash between steps 2 and 5 leaves a dangling `query_requests` row with no response. Those rows are:
- Detectable via `SELECT qr.id FROM query_requests qr LEFT JOIN query_responses qrs ON qrs.request_id = qr.id WHERE qrs.id IS NULL` for debugging.
- Excluded from conversation history fetches (which use an inner join).
- Safe to keep indefinitely; they are never served to any caller.

**History fetch SQL** (for reference — the actual implementation lives in `QueryResponse.fetch_recent_completed_turns`):

```sql
SELECT qr.question, qrs.answer, qrs.status
FROM query_requests qr
JOIN query_responses qrs ON qrs.request_id = qr.id
WHERE qr.session_id = $1 AND qr.conversation_id = $2
ORDER BY qr.created_at DESC
LIMIT $3  -- CONVERSATION_HISTORY_TURN_LIMIT, env, default 10
```

Results are reversed client-side to chronological order before being rendered into the prompt as `<turn_N>user: ... assistant: ...` blocks.

`feedback` — **at most one row per `query_id`** (replace semantics frozen in 2b).
| column | type | notes |
|---|---|---|
| id | `text` PK | `fbk_<random10>`. Stable across updates — the upsert preserves the row id. |
| query_id | `text` FK `query_requests.id` **on delete cascade**, **`UNIQUE`** | FK + UNIQUE means one row per `query_id`, cascade-deleted when the request is deleted. The `UNIQUE` index doubles as the conflict target for `ON CONFLICT (query_id)` in the upsert. Note the FK targets `query_requests`, not `query_responses` — feedback is keyed on the question, not the answer. |
| rating | `text` not null | Free-form (1–32 chars, non-blank, trimmed). Operator picks vocabulary. No CHECK constraint. |
| reason | `text` nullable | Free-form user text, ≤1000 chars. |
| created_at | `timestamptz` not null default `now()` | Timestamp of the *first* feedback event for this `query_id`. Preserved across replaces by the upsert. |
| updated_at | `timestamptz` not null default `now()` | Timestamp of the *latest* feedback event for this `query_id`. Advances on every replace. Equal to `created_at` until the first conflict. |

Indexes: unique on `query_id` (from the `UNIQUE` constraint above), plus `created_at` and `updated_at`.

**Action-layer upsert** (implemented in Batch 3c, documented here for cross-reference):

```sql
INSERT INTO feedback (id, query_id, rating, reason, created_at, updated_at)
VALUES ($1, $2, $3, $4, now(), now())
ON CONFLICT (query_id) DO UPDATE SET
    rating     = EXCLUDED.rating,
    reason     = EXCLUDED.reason,
    updated_at = now();
```

Only `rating`, `reason`, and `updated_at` are touched on conflict — `id` and `created_at` are preserved, so the row's PK is stable across replaces and `created_at` records when feedback was first given. The SQLAlchemy equivalent uses `sqlalchemy.dialects.postgresql.insert(...).on_conflict_do_update(...)`. Atomic and race-safe — no retry logic needed even under concurrent POSTs for the same `query_id`.

**Resolved in 2c (2026-04-14):**
- **Queries are split into two tables — `query_requests` and `query_responses`.** Written by 3b in two phases (request first, response after pipeline). Keeps this project as clean boilerplate for forkers who want their own response shape. Single-table was the original proposal and was reversed after design review: the two-table split is better for normalization, better for operators adopting the codebase as a base layer, and better for detecting crashed requests via dangling rows.
- **Conversation support added.** `query_requests` carries `session_id` (caller-supplied, required) and `conversation_id` (server-minted on omission/mismatch). History fetched server-side via a join on `query_responses`, capped by the `CONVERSATION_HISTORY_TURN_LIMIT` env var (default 10). See the [2b re-freeze note](./2b-contracts-and-schemas.md#status-of-this-batch) for the full contract delta.
- **Identifier format: random short strings.** `db/models/base.py` keeps the existing `_generate_random_string(10)` helper — 10 lowercase alphanumerics appended to the prefix. Resulting ids look like `doc_a3f8k2m9p1`. Readable in logs and CLI output, short enough to paste, collision-safe at v0 scale (36^10 per prefix). No new dependency. Id prefixes: `doc_` documents, `chk_` chunks, `qry_` query_requests (exposed as `query_id`), `qrs_` query_responses (internal), `fbk_` feedback, `cnv_` conversations (minted on `query_requests.conversation_id`).
- **`documents.source_url` is `UNIQUE`** — re-ingesting the same URL updates the existing row in place (new `content_hash`, chunks replaced via FK cascade). No history preserved in v0.
- **`chunks.token_count` dropped** — not read by any v0 code path. Add later if an eval harness needs it.
- **`queries.top_k` dropped** — `top_k` is a server-owned constant per deploy and is not part of the request contract.
- **`query_responses.latency_ms` dropped** — per-request latency is observability data, not domain data. Lives in logs/APM. Add back as a nullable column if a real eval harness needs it.
- **No `conversations` table.** `conversation_id` is just an indexed text column on `query_requests` (and denormalized onto `query_responses`). One fewer table, one fewer FK.
- **`query_responses.next_actions` added** as `jsonb not null default '[]'`. Stores the suggested follow-up prompts returned to the caller (see 2b re-freeze note). JSONB rather than `text[]` so we can evolve each entry into a richer object later (e.g. `{"prompt": ..., "kind": "related"}`) without a schema migration. No count cap at the DB layer — the prompt enforces the upper bound. No index in v0 — not queried directly.

**Resolved in 2b (carried over for the record):**
- `feedback.query_id` is a **required FK with cascade**. Orphan feedback is not allowed. Privacy wipes should use a scheduled job that nulls `question`/`answer` on the `queries` row but keeps the FK intact.
- `feedback.query_id` is also **`UNIQUE`** — one row per query (replace semantics).
- `feedback.rating` is free-form text, 1–32 chars, no CHECK constraint.
