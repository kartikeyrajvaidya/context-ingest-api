# Changelog

All notable changes to ContextIngest API are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- Public documentation (Batch 3f): `README.md` and `ARCHITECTURE.md` refreshed to match shipped behavior. New architecture docs: `docs/architecture/{overview,schema,ingestion-pipeline,retrieval}.md`. New guides: `docs/guides/{quickstart,ingestion,tuning-retrieval,self-hosting}.md`. API contract for ingest: `docs/api/ingest.md`. No code changes.
- Ingest endpoint (Batch 3h): `POST /v1/ingest` — manifest-driven ingestion over HTTP. No request body, reads `data/sources.json`, rerun-safe via content-hash comparison. Returns `{"data": {"ok": true}}` on success, 500 on failure. All per-entry operational detail (document IDs, chunk counts, errors) goes to server logs only — the HTTP response is deliberately minimal for a public API. Shared action in `core/actions/ingest_manifest.py` used by both the HTTP route and the CLI script (`scripts/ingest_all.py`). Not rate-limited (content-hash gate makes repeated calls free). Contract doc at `docs/api/ingest.md`.
- Safety classifier (Batch 3g): two-layer pre-LLM safety gate (regex heuristics + cheap LLM classifier), `SAFETY_ENABLED` toggle (default true), `SAFETY_LLM_MODEL` (default `gpt-4o-mini`). Detects prompt injection and jailbreak attempts. Fail-closed: classifier errors block the request. Client-facing `status` field on `QueryResponseSchema` (`answered|no_answer|refused|error`). `response_payload` JSONB consolidation on `query_responses` table (replaces five individual columns with a single JSONB column).
- Reembed script (Batch 3e): `scripts/reembed_all.py` walks every row in `chunks`, calls the OpenAI embeddings API with the existing `chunk_text`, and bulk-writes the new vectors in place. One-shot operator CLI for the same-dimension `OPENAI_EMBEDDING_MODEL` upgrade path — never re-fetches sources, never re-chunks. Keyset pagination via new `Chunk.iter_for_reembed(batch_size)` (yields `(id, chunk_text)` tuples, never loads the full table), bulk writes via new `Chunk.update_embeddings_bulk(updates)` (single `UPDATE ... FROM (VALUES ...)` per batch), and `Chunk.count_all()` for the progress denominator. Supports `--batch-size` (default 50) and `--dry-run`. One `commit_transaction_async()` block per batch so a crash mid-run leaves completed batches committed; the next run picks up idempotently. Cross-dimension upgrades remain out of scope — the `chunks.embedding` column is `Vector(1536)`.
- Rate limiting (Batch 3d): three stacked layers gate `/v1/query` and `/v1/feedback`. Layer 1 — per-IP per minute (default 5, `RATE_LIMIT_IP_PER_MINUTE`) — is an in-memory sliding window guarded by a `threading.Lock` with periodic prune, lives in `core/rate_limit/ip_limiter.py`, runs in `api/server/middleware.py::IPRateLimitMiddleware`, is scoped to the two public prefixes (so `/health` is exempt), resolves the client IP via `cf-connecting-ip` → `X-Forwarded-For` → `request.client.host`, and emits a startup WARNING if `WEB_CONCURRENCY > 1` since the counter is per-process. Layer 2 — per-session per hour (default 15, `RATE_LIMIT_SESSION_PER_HOUR`) — and Layer 3 — per-conversation total turns (default 10, `RATE_LIMIT_CONVERSATION_MAX_TURNS`) — live in `core/rate_limit/session_limiter.py`, read `query_requests` via `QueryRequest.count_session_requests_since` and the new `QueryRequest.count_conversation_turns`, and run at the top of `api/routes/query.py` before `query_document(...)` so in-flight requests don't count against their own budget. Both return `str | None`; the route maps the string to `HTTPException(429, detail=<message>)`. Over-budget requests emit one WARNING per trip naming the layer; in-budget requests produce no log. `configs/rate_limit.py` is extended with the two new thresholds; `.env.example` lists all three.
- Feedback path (Batch 3c): `POST /v1/feedback` is fully wired — replaces the Batch 2a `501` stub with a real implementation. `core/actions/record_feedback.py` verifies the `query_id` exists in `query_requests`, then calls `Feedback.upsert_by_query_id` inside `commit_transaction_async()` for an atomic `INSERT ... ON CONFLICT (query_id) DO UPDATE` — one row per `query_id`, last write wins, with `id` and `created_at` preserved across replaces while `rating`, `reason`, and `updated_at` advance. Unknown `query_id` raises `QueryNotFoundError` in the action layer (no fastapi import in `core/`), mapped to HTTP 404 `{"message": "query not found"}` in `api/routes/feedback.py`. Success returns `204 No Content`. Feedback on no-answer queries is supported (the spec's most valuable signal for tuning the grounding threshold).
- Initial repository scaffold: `README.md`, `ARCHITECTURE.md`, `CONTRIBUTING.md`, `CHANGELOG.md`, `LICENSE` (MIT).
- Community health files: `CODE_OF_CONDUCT.md` (Contributor Covenant 2.1), `SECURITY.md` (private disclosure policy + 3-day ack SLA).
- GitHub templates under `.github/`: `PULL_REQUEST_TEMPLATE.md`, `ISSUE_TEMPLATE/bug_report.yml`, `ISSUE_TEMPLATE/feature_request.yml`, `ISSUE_TEMPLATE/config.yml`, `CODEOWNERS`.
- Dependency hygiene: `.github/dependabot.yml` (weekly updates for pip, github-actions, docker).
- CI: `.github/workflows/ci.yml` — Ruff lint + format check, bytecode smoke test on Python 3.11 and 3.12, Docker image build.
- Modern Python tooling: `pyproject.toml` (PEP 621 metadata + Ruff + mypy config), `.pre-commit-config.yaml` (Ruff + file hygiene hooks), `.editorconfig`, `.python-version` → `3.11`.
- Container infrastructure: `Dockerfile`, `docker-compose.yaml`, `requirements.txt`, `.env.example`, `.gitignore`.
- Layered directory layout: `api/`, `configs/`, `core/`, `db/`, `libs/`, `scripts/`, `docs/`, `skills/` (skeleton only — code lands in subsequent batches).
- Planned v0 HTTP endpoints: `POST /v1/query`, `POST /v1/feedback`, `GET /health`. Ingestion is CLI-only in v0 (`scripts/ingest_url.py`, `scripts/ingest_batch.py`); a manifest-driven `POST /v1/refresh` trigger is documented in `docs/scaffolding-plan.md` as a future, auth-free-safe extension.
- Planned ingestion pipeline: fetch → clean → chunk → embed → persist, rerun-safe via SHA-256 content hash.
- Planned hybrid retrieval: pgvector + Postgres full-text fused with Reciprocal Rank Fusion.
- Planned repo-local Claude Code skills under `skills/context-ingest-api-backend/` and `skills/context-ingest-rag-pipeline/`.
- Query path (Batch 3b): `POST /v1/query` is fully wired — hybrid retrieval (pgvector + tsvector + RRF via `Chunk.hybrid_search`), single-call structured-output answer generation with `LLMAnswer` schema, 0-3 grounded `next_actions` follow-ups, opt-in multi-turn conversations via `session_id` + `conversation_id` (silent re-mint on stale ids), two-write durable flow (`query_requests` → retrieval + LLM → `query_responses`), and conversation history capped at `CONVERSATION_HISTORY_TURN_LIMIT` (default 10). New modules: `core/schema/llm_answer.py`, `core/services/{openai_client,prompts,llm,retrieval}.py`, `core/actions/query_document.py`. `core/services/embeddings.py` migrated off its inline `AsyncOpenAI(...)` to the shared `openai_client.get_openai_client()` factory.
- Ingestion pipeline (Batch 3a, CLI-only): `core/services/embeddings.py` (OpenAI embeddings adapter with sub-batch splitting), `core/ingestion/{fetcher,cleaner,chunker,pipeline}.py` (httpx fetcher, trafilatura + markdown cleaner, token-bounded recursive chunker with overlap, rerun-safe hash/chunk/embed/persist pipeline), `core/actions/ingest_document.py` (single action owning the transaction boundary and mapping non-fatal failures to `status: "failed"`), and `scripts/ingest_url.py` + `scripts/ingest_batch.py` for single-URL and newline-list ingestion.
- Backend skeleton (Batch 2): env-driven config (`configs/{common,db,llm,rate_limit}.py`), async SQLAlchemy engine + scoped session (`db/connections/context_ingest_db.py`), declarative models for `documents`, `chunks` (with pgvector + tsvector), `queries`, `feedback`, raw-SQL + Alembic init migration with HNSW and GIN indexes, FastAPI boot module with CORS, IP rate-limit middleware (no-op until Batch 3c), error handlers, request-validation helper for the `{"data": {...}}` envelope, working `GET /health` route, `501` stubs for `/v1/query` and `/v1/feedback`, and Pydantic schemas under `core/schema/`. (Batch 2 also shipped a `501` stub for `/v1/ingest`; that stub is removed in Batch 3a, which makes ingestion CLI-only.)
